## Overview
- Priority: P1
- Current status: pending
- Description: finish the unfinished execution path for Phase 5 by moving from stored settings and preview screens to real runtime response routing, usable backfeed, configurable session intelligence, and persona consistency checks.

## Landed Baseline
- `apps/api` already persists per-node response config and audits updates.
- `apps/node` fetches node config and persona data, but protocol services still call template routing directly.
- `packages/response-engine` supports keyword overlap template matching, static fallback, and streaming chunk splitting only.
- `apps/worker` already classifies completed sessions, enriches IPs, correlates actors, and materializes alert logs.
- `apps/web` already exposes response-config preview on node detail plus read-only sessions, personas, and actors views.

## Remaining Scope
- Execute `smart`, `fixed_n`, and `budget` strategies inside the node runtime instead of only storing them.
- Support generic OpenAI-compatible proxy routing with encrypted secret handling and budget accounting.
- Turn uncovered prompts into a reviewable backfeed queue using existing template records instead of a placeholder flag.
- Replace hard-coded classifier and enrichment helpers with configurable, persisted inputs and real provider adapters.
- Add persona consistency validation so dashboard edits can detect contradictions before a persona ships to nodes.
- Expand operator UI from previews to edits, review actions, drill-downs, and replay.

## Data Flows
1. Operator updates response config in the dashboard. API validates the payload, encrypts the proxy secret at rest, and serves a decrypted runtime config only on the node config endpoint.
2. Node refreshes config, compiles the strategy chain, and routes each LLM request through a shared orchestrator that decides `real_model`, `template`, or safe fallback before capture persistence.
3. When no approved template covers a prompt, the orchestrator emits a backfeed candidate keyed by `promptHash`; API or worker deduplicates it against `ResponseTemplate`, stores it as `autoGenerated=true` and `approved=false`, and exposes it to an operator review queue.
4. On session completion, worker loads the active rule pack, classifies the session, enriches the source IP, updates related requests, and re-runs actor scoring with the new metadata.
5. Persona create or update flows run a consistency validator before save and again before node delivery so protocol-specific bait surfaces stay aligned.

## Implementation Steps
1. Harden the response-config contract by masking or encrypting `proxy.apiKey`, keeping `Node.config.responseConfig` as the stored source of truth, and serving a runtime-safe representation to nodes.
2. Split `packages/response-engine` into small modules for prompt inspection, template selection, budget checks, proxy calls, and backfeed candidate emission.
3. Update node protocol services so every LLM response path flows through a single orchestrator instead of direct `routeTemplateResponse` calls.
4. Reuse `BudgetEntry` for per-node or global monthly accounting and record actual proxy decisions in capture metadata.
5. Reuse `ResponseTemplate` for the backfeed review queue by storing generated prompts and responses as `autoGenerated=true` and `approved=false`, with dedup keyed by `promptHash`.
6. Replace hard-coded classifier arrays and synthetic enrichment builders with loadable rule packs and provider adapters, while preserving offline-safe fallback behavior for local development.
7. Add persona consistency checks that validate model, hardware, config-file, and shell-surface claims before save and before node delivery.
8. Expand the dashboard UI from preview-only screens to editable response config, backfeed review, persona validation, and session replay workflows.

## Risk Assessment
- High likelihood, high impact: proxy outages or slow models make the honeypot obviously fake. Mitigation: timeout, retry cap, template fallback, and capture of provider error reason.
- Medium likelihood, high impact: budget tracking races overspend on burst traffic. Mitigation: one transactional write path per proxy decision.
- Medium likelihood, medium impact: operator-configurable rules create untestable combinations. Mitigation: validate rule packs at write time and ship defaults first.
- Medium likelihood, medium impact: persona validation blocks current presets. Mitigation: validate built-ins first and surface actionable errors instead of silent mutation.

## Backwards Compatibility
- Preserve current API route paths and default response-config values.
- Preserve existing template-only behavior when proxy settings are empty or disabled.
- Preserve current worker classification and enrichment outputs as fallback defaults until replacements are enabled.

## Rollback Plan
- Disable proxy or backfeed through response config and redeploy to return to template-only runtime behavior.
- Revert to bundled classifier and synthetic enrichment helpers if provider-backed processors fail.
- Leave existing web read routes intact so operators retain visibility even if edit workflows are rolled back.

## Success Criteria
- Every LLM protocol path goes through one shared orchestrator and records the chosen strategy in captures.
- Proxy secrets are no longer stored or returned in plaintext.
- Backfeed candidates can be reviewed and approved without custom DB tables beyond what already exists.
- Classification and enrichment produce persisted outputs from configurable inputs instead of only static arrays or synthetic data.
- Dashboard UI can edit response config, review backfeed, replay sessions, and validate personas.
